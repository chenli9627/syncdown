基于AI大模型的多人协作文档系统的设计与实现

摘 要

随着远程办公、在线教育和团队知识管理场景的不断增长，传统单机文档编辑方式已经难以满足多人实时协同、结构化内容管理、权限隔离以及智能化写作辅助等需求。针对上述问题，本文设计并实现了一款基于 AI 大模型的多人协作文档系统 Syncdown。系统以 Web 文档编辑为核心场景，在满足富文本编辑、多人协同、版本记录和资源管理等基础能力的同时，将大语言模型接入到文档编辑流程中，使系统不仅能够完成多人共同编辑，还能够围绕选中文本提供翻译、总结、解释和润色等智能辅助能力。

系统采用 Next.js、React 与 TypeScript 构建前端与服务端接口，使用 Tiptap 作为结构化富文本编辑器，基于 Yjs 与 Hocuspocus 实现多人协同编辑。系统将用户编辑操作转换为 Y.Doc 中的 CRDT 增量更新，并通过 WebSocket 按文档房间进行广播；各客户端在本地合并更新后保持最终一致。针对 AI 大模型接入问题，系统采用服务端代理方式封装 OpenAI 兼容接口，前端仅提交选中文本、操作类型、语言环境和候选数量等上下文，模型密钥与模型调用过程均由服务端统一管理。用户可以在主模型和备选模型生成的候选结果中进行选择，并将结果替换选区或插入到文档指定位置，从而实现“AI 参与写作但用户确认落稿”的交互流程。

在系统设计方面，本文围绕工作区权限、文档共享、协同编辑、AI 辅助、媒体资源管理和版本追踪等模块展开分析，并对 CRDT 增量同步、Awareness 状态同步、自动保存节流、AI 候选结果生成与回填等关键过程进行了数学化描述。系统支持 PostgreSQL 与本地 JSON 两种状态持久化方式，并支持本地文件系统与 S3 兼容对象存储，以提高部署灵活性。测试结果表明，系统能够完成用户认证、文档管理、多人协同编辑、权限控制、Markdown 导入导出、图片上传、版本历史、更新记录以及 AI 多模型辅助等功能，在毕业设计实验环境下运行稳定，能够较好地满足中小团队协作文档场景的应用需求。

关键词：AI大模型；多人协作文档；富文本编辑；CRDT；Yjs；Next.js；Tiptap

Design and Implementation of a Multi-User Collaborative Document System Based on AI Large Model

**ABSTRACT**

With the increasing demand for remote work, online education, and team knowledge management, traditional standalone document editors can no longer meet the requirements of real-time multi-user collaboration, structured content management, permission isolation, and intelligent writing assistance. To address these problems, this thesis designs and implements Syncdown, a multi-user collaborative document system based on AI large models. The system focuses on Web-based document editing and integrates large language model capabilities into the editing workflow while providing rich-text editing, collaboration, version tracking, and resource management.

The system is developed with Next.js, React, and TypeScript, uses Tiptap as the structured rich-text editor, and adopts Yjs with Hocuspocus to implement real-time collaborative editing. Editing operations are transformed into CRDT updates in Y.Doc and broadcast to clients in the same document room through WebSocket. Each client merges the updates locally and eventually reaches a consistent document state. For AI integration, the system uses a server-side proxy to call OpenAI-compatible model APIs. The frontend submits only the selected text, operation type, locale, and candidate count, while model keys and external API requests are managed on the server side. Users can compare candidates generated by the primary and secondary models and then replace the selected text or insert the result into the document, forming an interaction pattern in which AI assists writing and users confirm the final content.

In the system design, this thesis analyzes workspace permissions, document sharing, collaborative editing, AI assistance, media resource management, and version tracking. It also describes the key processes of CRDT update synchronization, Awareness state synchronization, autosave throttling, AI candidate generation, and result application in mathematical form. The system supports both PostgreSQL and local JSON persistence, as well as local file storage and S3-compatible object storage. Test results show that the system can complete user authentication, document management, real-time collaboration, permission control, Markdown import and export, image uploading, version history, update records, and AI multi-model assistance. In the graduation-design experimental environment, the system runs stably and satisfies the basic requirements of small and medium team collaborative document editing.

**Key words：**AI Large Model; Collaborative Document System; Rich-text Editing; CRDT; Yjs; Next.js; Tiptap

# 第1章 绪论

## 1.1 研究背景与意义

随着互联网技术的快速发展以及信息化水平的不断提高，基于Web的应用系统已广泛应用于办公、教育及企业协作等多个领域。特别是在远程办公和在线协作日益普及的背景下，传统的本地文档编辑方式逐渐暴露出诸多不足，如难以实现多用户实时协同、数据共享效率低以及跨平台支持能力有限等问题。因此，设计并实现一种支持多用户实时协作的在线文档系统具有重要的现实意义。

近年来，前端技术的快速发展为复杂Web应用的实现提供了有力支撑。以React为代表的前端框架通过组件化开发模式，有效提升了系统的可维护性与扩展性^\[1\]\[3\]^。同时，前后端分离架构的广泛应用，使得系统开发更加灵活高效，能够更好地适应复杂业务需求^\[15\]^。在数据交互与系统性能方面，Node.js等技术为高并发处理提供了良好的支持^\[4\]^。

在协同编辑领域，基于CRDT（Conflict-free Replicated Data
Type）思想的实时同步技术逐渐成为研究热点。通过引入如Yjs等协同算法，可以在多用户同时编辑的场景下实现数据的一致性与实时同步，从而显著提升系统的协作效率。此外，随着大语言模型技术的发展，AI辅助文本编辑逐渐成为文档系统的重要功能之一，为用户提供智能化的写作支持与内容优化能力^\[6\]\[7\]^。

因此，研究并实现一个集富文本编辑、实时协同、权限管理与AI辅助于一体的协作文档系统，不仅能够满足实际应用需求，还具有一定的研究价值和应用前景。

## **1.2 国内外研究现状**

国外在线协作文档系统起步较早，Google Docs、Microsoft 365、Notion 等产品已经形成较成熟的多人编辑、权限共享、历史版本和内容组织能力。在协同编辑算法方面，早期系统多采用 OT（Operational Transformation）处理并发编辑冲突，近年来 CRDT（Conflict-free Replicated Data Type）因其天然支持离线合并、乱序更新和最终一致性，逐渐成为 Web 协同编辑的重要技术路线。Yjs、Automerge 等开源项目降低了 CRDT 在实际工程中的应用成本，使开发者能够在浏览器端构建可同步的共享文档模型。

随着大语言模型技术的发展，国外主流文档产品开始将 AI 能力嵌入文档编辑流程。Notion AI、Microsoft Copilot 和 Google Workspace AI 等工具不再只提供普通文本输入，而是围绕“选中文本-提出指令-生成候选-用户确认”的交互过程，提供摘要、改写、翻译、语气调整和内容扩写等能力。这类系统的特点是 AI 不直接取代用户编辑，而是作为写作辅助层参与文档生成与优化。

国内在线协同办公与知识管理产品近年来发展较快，腾讯文档、飞书文档、语雀等产品在多人协作、权限管理和文档结构化方面已有广泛应用。相关研究也逐渐从传统的文档管理系统转向基于 Web 的实时协作系统，常见技术路线包括前后端分离架构、组件化前端框架、WebSocket 实时通信和关系型数据库持久化。在 AI 应用方面，国内大模型服务逐步成熟，文档系统开始探索将文本生成、润色、翻译和知识问答能力接入编辑器，但在毕业设计和中小型系统实现中，仍常见 AI 功能与文档编辑流程割裂、协同编辑与 AI 回填缺少统一事务处理的问题。

综上，现有研究和产品已经分别在协同编辑、富文本编辑和 AI 写作辅助方面取得较好成果，但仍存在三个值得继续研究的问题：第一，如何在中小规模 Web 系统中以较低复杂度实现多人实时协同；第二，如何在权限、版本、媒体资源和导入导出等工程功能之间保持一致的数据模型；第三，如何将 AI 大模型以可控、可确认的方式嵌入文档编辑流程，使 AI 生成结果能够成为普通编辑事务并进入协同同步链路。本文围绕上述问题设计并实现 Syncdown 系统。

## 1.3 研究内容

本文围绕基于 AI 大模型的多人协作文档系统设计与实现展开研究，主要内容包括：

（1）需求分析：结合团队协作文档场景，分析系统在用户认证、工作区管理、文档共享、富文本编辑、协同编辑、版本记录、媒体资源管理和 AI 辅助写作等方面的功能需求，并给出性能、可靠性、安全性和可扩展性等非功能需求。

（2）系统总体设计：基于 Next.js、React、TypeScript、Tiptap、Yjs、Hocuspocus、PostgreSQL 和对象存储等技术，设计系统总体架构、功能模块、数据组织方式和权限模型，明确前端展示层、业务接口层、协同编辑层、状态存储层、媒体存储层和 AI 服务层之间的关系。

（3）核心算法与流程设计：对协同编辑过程中的 CRDT 增量同步、状态向量合并、Awareness 协作者状态同步、自动保存节流、版本快照生成以及 AI 候选结果生成与回填过程进行形式化描述，使用数学符号说明系统如何保证并发编辑最终一致和 AI 结果可控落稿。

（4）功能模块实现：实现用户认证、工作区与权限管理、文档生命周期管理、Tiptap 富文本编辑、Yjs 实时协同、Markdown 导入导出、图片上传、版本历史、更新记录和 AI 多模型辅助编辑等模块。

（5）系统测试与结果分析：通过功能测试、协同场景测试、权限测试、AI 功能测试和性能观察，验证系统核心链路的正确性与可用性，并结合测试结果分析系统优点、不足和后续改进方向。

## 2.1 Next.js

本系统采用 Next.js 作为 Web 应用框架，前端界面使用 React 和 TypeScript
实现。选择该技术组合的主要原因在于：一方面，React
的组件模型适合将工作区侧栏、文档编辑器、权限弹窗、AI
气泡等复杂界面拆分为相互独立的功能组件；另一方面，Next.js
能够在同一工程中组织页面路由和服务端
API，使文档保存、媒体上传、权限变更等接口与前端页面保持清晰的模块边界。^\[1\]\[3\]\[13\]\[15\]^

TypeScript
在本系统中主要用于约束用户、工作区、文档、权限和版本记录等核心数据结构。由于
Syncdown 存在 owner、guest、can_edit、can_view
等权限分支，如果缺乏明确类型约束，前端组件和服务端 mutation
很容易出现权限状态不一致的问题。因此，TypeScript
的作用不是简单提高语法安全性，而是帮助系统在多人协作和权限控制场景下保持数据结构一致。^\[8\]\[9\]^

## 2.2 Tiptap

Syncdown 的正文编辑能力基于 Tiptap 实现。Tiptap 底层使用 ProseMirror
文档模型，能够将编辑器内容表示为结构化节点，因此适合实现标题、段落、列表、代码块、图片、链接和表格等富文本能力。在本系统中，Tiptap
不仅负责文本输入和格式渲染，还承担块级菜单、Slash
菜单、图片拖放上传、选区 AI 气泡等交互入口。

由于系统需要支持实时协同编辑，普通 HTML
文本框无法满足并发修改和结构化同步要求。Tiptap 的 Collaboration
扩展可以将编辑器文档绑定到 Yjs 的
Y.Doc，使用户的本地编辑操作能够转换为可同步的 CRDT
更新。这也是本系统选择 Tiptap 而不是普通富文本控件的主要原因。

## 2.3 Yjs

本系统采用 Yjs 作为协同编辑引擎，采用 Hocuspocus 作为 WebSocket
协作服务。Yjs 提供基于 CRDT
的共享文档模型，负责处理多人并发编辑时的更新合并和最终一致性；Hocuspocus
负责维护客户端连接，并将同一文档房间内产生的 update 广播给其他客户端。

需要说明的是，本系统并未自行实现 CRDT 底层数据结构，而是在应用层集成 Yjs
提供的协同能力。系统自身完成的工作包括：按文档 ID 创建协作房间、将初始
HTML 内容写入空的 Y.Doc、把 Tiptap 编辑器绑定到 Y.Doc、利用 Awareness
提取协作者状态、将协同编辑后的 HTML
内容保存到应用状态中。这样的设计降低了自研同步算法的复杂度，同时保证了协同编辑功能的可靠性。

## 2.4 PostgreSQL和MinIO

系统状态存储采用 PostgreSQL 与本地 JSON 回退机制。配置 DATABASE_URL
时，用户、工作区、文档、权限关系、版本历史等应用状态写入
PostgreSQL；未配置数据库时，系统回退到
.data/app-state.json，以便开发和演示环境快速运行。媒体资源则通过本地存储或
S3 兼容对象存储保存，编辑器和头像面向用户的访问路径统一保持为
/api/media/\...，避免正文内容直接依赖具体桶地址。另外，通过MinIO作为对象存储来存储图片这样的大文件。^\[10\]\[11\]\[12\]^

## 2.5 AI 辅助编辑技术

AI 辅助功能采用服务端代理方式集成 OpenAI
兼容接口。前端只负责采集编辑器选区、展示 AI
操作气泡、设置候选结果数量并回填结果，模型密钥读取和外部接口调用均在服务端完成。系统支持单模型生成和双模型结果对比：当用户开启结果对比时，服务端会同时请求主模型和备选模型，并把候选结果连同模型名称返回前端，由用户选择其中一个结果应用到正文中。^\[6\]\[7\]^

## 2.6 本章小结

本章从系统职责出发介绍了 Syncdown 使用的关键技术。Next.js、React 与
TypeScript 负责页面和接口组织，Tiptap 负责结构化富文本编辑，Yjs 与
Hocuspocus 负责实时协同，PostgreSQL、JSON 与对象存储负责持久化，AI
服务端代理负责智能编辑能力。这些技术共同服务于多人协作文档系统的核心目标。

# 第3章 系统分析

## 3.1 应用场景分析

Syncdown
面向团队协作文档场景，目标用户包括需要统一管理文档的工作区拥有者，以及通过共享关系进入工作区并参与协作的
guest
用户。与普通单机文档编辑工具相比，该系统更强调多人同时编辑、权限边界清晰、富文本结构化表达、媒体资源统一管理以及
AI 辅助处理等能力。

在实际使用过程中，用户通常需要在同一工作区内完成文档创建、共享、协同编辑、版本回溯、Markdown
导入导出和图片插入等操作。系统因此不能只解决"能不能编辑文本"的问题，还需要解决"谁能编辑""如何避免并发冲突""如何保存历史状态""AI
结果如何安全地回填正文"等一系列业务问题。

## 3.2 功能需求分析

本系统面向多人协作文档场景，功能需求按照“用户与权限、文档编辑、协同同步、版本追踪、资源管理、AI 辅助”六类展开。

（1）用户与工作区权限需求：系统应支持用户注册、登录和基本账号信息管理；支持用户创建工作区，并在工作区内区分 owner 与 guest。owner 可以创建文档、管理共享关系、修改标题、移动回收站和恢复文档；guest 只能访问被显式共享的文档，并根据 can_edit 或 can_view 权限决定是否可编辑正文。

（2）文档编辑需求：系统应提供结构化富文本编辑能力，支持段落、一级至六级标题、无序列表、有序列表、任务列表、引用、代码块、表格、链接、删除线、图片等常用文档元素；同时支持 Slash 菜单、块级菜单、拖拽交互、Markdown 导入导出和 ZIP 资源打包导出。

（3）实时协同需求：系统应允许多个用户同时进入同一文档编辑。任一客户端产生的正文修改都应通过协作服务同步到其他客户端，并在并发插入、并发删除、网络乱序和重复更新情况下保持最终一致。系统还应提供协作者在线状态和编辑位置提示，使用户能够感知其他协作者正在编辑的区域。

（4）历史追踪需求：系统应记录文档修改过程，提供版本历史和更新记录。版本历史用于查看和恢复历史快照，更新记录用于展示最近修改人、修改时间和文本差异。普通自动保存与可恢复版本快照应分开控制，避免每次输入都生成过多历史版本。

（5）文件与媒体需求：系统应支持图片上传、头像访问和文档资源导出。编辑器正文中的媒体资源不应直接保存外部桶地址，而应统一保存 /api/media/... 应用路径，由服务端根据当前存储后端读取本地文件或 S3 兼容对象。

（6）AI 辅助编辑需求：系统应支持用户选中文本后调用 AI 大模型进行翻译、总结、解释、润色和自定义提示处理；支持主模型和备选模型候选结果对比；AI 返回结果需要由用户确认后才能替换选区或插入下方。AI 功能应通过服务端代理调用模型接口，前端不得暴露模型密钥。

## 3.3 非功能需求分析

除功能需求外，系统还需满足以下非功能性要求：

（1）性能要求：系统应具备良好的响应速度，在多用户协同编辑场景下仍能保持较低延迟。

（2）可靠性要求：系统应具备数据持久化能力，避免数据丢失，并支持异常情况下的数据恢复。

（3）可扩展性要求：系统应具备良好的模块化结构，便于后续功能扩展与系统升级。

（4）安全性要求：系统应保证用户数据的安全性，防止未授权访问与数据泄露。

（5）可用性要求：系统界面应简洁易用，操作流程清晰，提升用户体验。

## 3.4 角色与权限分析

Syncdown 的角色不是全局固定角色，而是与 workspace 绑定的相对身份。创建
workspace 的用户在该工作区内是 owner，被共享文档带入该工作区的用户是
guest。同一用户可以在某个 workspace 中是 owner，在另一个 workspace 中是
guest。

在文档权限层面，owner
拥有创建文档、修改标题、管理共享关系、移动回收站和恢复删除文档等权限；guest
只能访问明确分享给自己的文档，并根据 can_edit 或 can_view
权限决定是否允许编辑正文。该权限模型直接影响侧栏可见文档、编辑器可写状态、服务端保存入口以及版本恢复入口。

## 3.5 关键问题分析

系统实现过程中需要重点解决以下关键问题。

第一，多人同时编辑同一文档时，如果采用普通“读取-修改-覆盖保存”的方式，后提交的内容可能覆盖先提交的内容，导致数据丢失。因此系统需要引入 CRDT 协同机制，将编辑操作表示为可交换、可合并的增量更新，使不同客户端在接收同一组更新后最终收敛到一致状态。

第二，协同编辑状态与应用持久化状态需要分层处理。Y.Doc 负责实时冲突合并和客户端同步，PostgreSQL 或本地 JSON 负责保存系统状态。如果每次按键都直接生成版本快照，会造成历史数据膨胀；如果只保存最终 HTML，又会丢失可恢复历史。因此系统需要同时设计自动保存、版本快照和更新记录三条链路。

第三，工作区 owner、guest、can_edit 和 can_view 之间存在交叉关系。权限控制不能只依赖前端按钮显示，还必须在服务端保存、共享、回收站、版本恢复等入口进行统一校验，避免只读用户绕过界面限制写入数据。

第四，AI 大模型生成结果具有不确定性，且模型调用涉及密钥、网络延迟和候选结果质量问题。系统不能让 AI 自动覆盖用户文档，而应采用“选区上下文提交、服务端代理调用、候选结果展示、用户确认回填”的流程，使 AI 辅助结果作为普通编辑事务进入 Tiptap 与 Yjs 同步链路。

## 3.6 本章小结

本章围绕系统应用场景、功能需求、非功能需求、角色权限边界以及关键问题进行了分析，明确了后续系统设计与实现需要解决的核心目标，为后续章节展开系统设计提供了依据。

## 4.6 协同编辑算法设计

本文并未从零实现新的 CRDT 底层结构，而是在系统中集成 Yjs 提供的 CRDT 协同模型，并围绕文档初始化、增量同步、状态向量合并、协作者状态同步、自动保存和版本记录实现应用层算法流程。为了更清晰地描述系统协同机制，设某一文档在客户端 i 上的本地副本为 \(D_i\)，所有客户端副本集合为 \(\mathcal{D}=\{D_1,D_2,\ldots,D_n\}\)。文档内容由一组带唯一标识的结构化节点组成，如式(4-1)所示。

\[
D_i=(N_i,\prec_i),\quad N_i=\{(id,c,attr,del)\}
\tag{4-1}
\]

其中，\(id=(client,clock)\) 表示节点全局唯一标识，\(c\) 表示文本或结构化内容，\(attr\) 表示标题、加粗、列表、链接等格式属性，\(del\) 表示节点是否被删除，\(\prec_i\) 表示客户端 i 当前可见的节点偏序关系。一次本地编辑操作会生成增量更新 \(u\)，更新可表示为式(4-2)。

\[
u=(op,id,origin,rightOrigin,payload,t)
\tag{4-2}
\]

其中，\(op\) 表示插入、删除或属性修改；\(origin\) 与 \(rightOrigin\) 用于描述插入位置左右邻接关系；\(payload\) 表示操作内容；\(t\) 表示逻辑时间。客户端产生更新后，先在本地执行应用函数 \(apply\)，再通过 Hocuspocus 广播给同一文档房间内的其他客户端，如式(4-3)所示。

\[
D_i^{'}=apply(D_i,u),\quad broadcast(documentId,u)
\tag{4-3}
\]

当远端客户端收到更新集合 \(U=\{u_1,u_2,\ldots,u_k\}\) 时，Yjs 根据 CRDT 合并规则执行可交换合并。若任意两个客户端最终接收到相同更新集合，则它们的最终文档状态满足式(4-4)。

\[
\forall i,j,\quad U_i=U_j \Rightarrow merge(D_i,U_i)=merge(D_j,U_j)
\tag{4-4}
\]

式(4-4)说明系统的一致性不依赖服务端裁决顺序，而依赖 CRDT 更新的幂等性、交换性和可合并性。对于重复更新，Yjs 通过状态向量过滤已应用的逻辑时钟；对于乱序更新，Yjs 保留 origin 与 rightOrigin 等位置引用，待依赖更新到达后完成正确定位。状态向量可表示为式(4-5)。

\[
SV_i=\{clientId \mapsto clock_{max}\}
\tag{4-5}
\]

客户端重连时只需根据状态向量请求缺失更新集合 \(\Delta U\)，如式(4-6)所示。

\[
\Delta U=U_{server}-U(SV_i)
\tag{4-6}
\]

除正文 CRDT 同步外，系统还使用 Awareness 同步协作者状态。设用户 k 的在线状态为 \(A_k=(userId,documentId,blockId,color,updatedAt)\)，协作者状态集合为 \(A=\{A_1,A_2,\ldots,A_m\}\)。Awareness 状态只用于界面提示，不写入正文 CRDT，也不生成版本历史。其合并规则取同一用户最新时间戳状态，如式(4-7)所示。

\[
A(userId)=\arg\max_{A_k.userId=userId} A_k.updatedAt
\tag{4-7}
\]

在持久化方面，系统将实时同步与自动保存分开。设用户连续编辑产生的保存候选序列为 \(S=\{s_1,s_2,\ldots,s_m\}\)，自动保存延迟窗口为 \(\tau\)。当两次候选保存间隔小于 \(\tau\) 时，系统只保留最新状态，如式(4-8)所示。

\[
save(D,t)=
\begin{cases}
skip, & t-t_{last}<\tau \\\\
persist(HTML(D)), & t-t_{last}\geq \tau
\end{cases}
\tag{4-8}
\]

版本快照则采用独立策略生成。设第 k 次可恢复版本为 \(V_k=(html_k,author_k,time_k,reason_k)\)，当修改来源为手动保存、版本恢复或超过最小快照间隔时生成新版本，如式(4-9)所示。

\[
V_{k+1}=
\begin{cases}
(HTML(D),author,t,reason), & manual \lor restore \lor (t-t_{V_k}\geq \lambda) \\\\
V_k, & otherwise
\end{cases}
\tag{4-9}
\]

协同编辑算法流程如图4-5所示。该流程体现了正文 CRDT 同步、Awareness 状态同步和应用层保存三条路径：CRDT 负责实时一致性，Awareness 负责协作者可见性，自动保存与版本快照负责持久化和历史恢复。

## 4.7 AI 辅助流程设计

AI 辅助功能以编辑器选区为入口。用户选中文档中的一段内容后，系统计算选区位置并显示 AI 操作气泡。用户可以选择翻译、总结、解释、润色或输入自定义提示词；前端将选中文本、操作类型、语言环境、提示词和候选数量提交到服务端，由服务端读取模型配置并调用 OpenAI 兼容接口。

设用户选中文本为 \(x\)，操作类型为 \(a\)，用户补充提示词为 \(p\)，语言环境为 \(l\)，当前启用模型集合为 \(M=\{m_1,m_2,\ldots,m_r\}\)。服务端构造模型输入上下文 \(C\)，如式(4-10)所示。

\[
C=(x,a,p,l,systemPrompt)
\tag{4-10}
\]

对每一个模型 \(m_i\)，系统调用生成函数 \(F\) 得到候选结果 \(y_i\)，如式(4-11)所示。

\[
y_i=F(m_i,C),\quad i=1,2,\ldots,r
\tag{4-11}
\]

候选结果集合为 \(Y=\{(m_i,y_i)\}\)。系统不自动覆盖正文，而是将候选结果展示给用户，由用户选择其中一个结果 \(y^*\) 并指定应用方式。若应用方式为替换选区，则编辑器事务可表示为式(4-12)。

\[
D'=replace(D,range(x),y^*)
\tag{4-12}
\]

若应用方式为插入下方，则编辑器事务可表示为式(4-13)。

\[
D'=insertAfter(D,block(x),y^*)
\tag{4-13}
\]

无论采用替换还是插入，最终修改都会转换为一次普通 Tiptap 编辑事务，并进入 Y.Doc 生成 CRDT update，因此 AI 结果也会同步给其他协作者。该设计使 AI 大模型被嵌入文档编辑流程，而不是作为与正文无关的独立问答工具。AI 多模型辅助流程如图4-6所示。

# 第5章 系统实现

## 5.1 系统实现总体说明

本系统基于 Next.js
全栈框架进行开发，采用前后端一体化架构，将页面展示与后端逻辑统一在同一工程中实现。前端部分基于
React 与 TypeScript 实现组件化开发，后端通过 Next.js 提供的 API
路由机制处理业务请求。同时，引入 Yjs 与 Hocuspocus
实现实时协同编辑功能，并结合 PostgreSQL
与对象存储技术实现数据持久化与资源管理。

## 5.2 用户认证模块实现

用户认证模块负责用户注册、登录、身份校验和基础资料维护。系统通过表单组件收集邮箱、用户名、昵称和密码等信息，服务端负责合法性校验、用户唯一性判断、密码哈希处理以及登录态维持。该模块为后续工作区切换、文档访问与权限判定提供前置身份基础。

## 5.3 文档管理与权限实现

文档管理模块不仅负责文档列表展示，还承担工作区边界、共享状态和回收站生命周期管理。系统中的文档默认属于某个
workspace，workspace owner 创建的未分享文档归入 Private
区；当文档被分享给 guest 后，文档会出现在 Shared 区；当所有 guest
访问关系被移除后，文档重新回到 Private 区。

在实现过程中，前端侧栏根据当前用户、当前
workspace、文档列表和访问关系计算可见文档集合。owner 进入自己的
workspace 时可以看到 Private、Shared 和 Trash；guest 进入被分享的
workspace
时，只能看到明确分享给自己的文档。服务端在文档相关接口中再次执行权限校验，从而避免前端异常导致的越权写入。

## 5.4 富文本编辑模块实现

富文本编辑模块的实现过程主要分为编辑器实例构建、扩展配置、权限控制和内容保存四个步骤。首先，系统在文档页面中创建
Tiptap 编辑器实例，并根据当前文档的协同状态决定是否使用本地 HTML
内容初始化编辑器。当协同文档存在时，编辑器内容不直接从普通 HTML
初始化，而是交由 Collaboration 扩展从 Y.Doc
中读取，避免本地内容覆盖远程协同状态。

其次，系统通过扩展机制配置图片、链接、代码块、任务列表、表格和目录等能力。图片处理不是简单插入
base64 字符串，而是通过上传接口写入媒体存储，再将返回的 /api/media/\...
地址插入编辑器。最后，系统在编辑器 onUpdate 事件中触发自动保存，并通过
500ms 防抖合并短时间内的连续修改。

## 5.5 协同编辑模块实现

每个客户端进入文档页面后，系统首先根据当前文档 ID 创建一个新的
Y.Doc，并通过 HocuspocusProvider 连接协作服务器。协作房间名称采用
document-{documentId} 的形式，使不同文档的协同数据相互隔离。随后，Tiptap
Collaboration 扩展将编辑器文档绑定到 Y.Doc 的 default
字段。用户在编辑器中产生的正文变更不再只存在于本地 React
状态中，而是写入 Y.Doc，由 Yjs 转换为 CRDT update。

协同编辑房间初始化代码如下：

const doc = new Y.Doc();

const provider = new HocuspocusProvider({

document: doc,

name: \`document-\${documentId}\`,

onSynced: () =\> setCollaborationSynced(true),

url: getCollabServerUrl(),

});

setCollaborationProvider(provider);

setCollaborationDocument(doc);

上述代码展示了系统进入文档页面后如何创建独立的 Y.Doc，并使用 documentId
生成 Hocuspocus
协作房间名称。该房间名称是协同数据隔离的关键，不同文档对应不同房间，因此同一文档内的
CRDT update 只会广播给正在编辑该文档的客户端。

Tiptap 编辑器绑定 Y.Doc 的关键代码如下：

Collaboration.configure({

document: collaborationDocument,

field: \"default\",

}),

CollaborationCaret.configure({

provider: collaborationProvider,

user: { name: currentUser.name, userId: currentUser.id },

}),

content: collaborationDocument ? undefined : toEditorContent(content),

上述代码展示了协同模式下编辑器正文不再由普通 HTML 直接初始化，而是通过
Collaboration 扩展绑定到 Y.Doc 的 default
字段。这样用户输入首先进入共享文档模型，再由 Yjs
产生增量更新并同步到其他客户端。

初始内容处理是协同编辑实现中的一个关键细节。由于数据库中保存的是文档
HTML，而协同编辑运行时使用的是 Y.Doc，如果每个客户端进入页面后都直接调用
setContent，就可能重复覆盖协作文档。因此系统在同步完成后先检查 Y.Doc
中的 fragment 是否已有内容，并读取 config.initialContentLoaded
标记。只有当 fragment 为空、数据库内容非空且尚未初始化时，客户端才会将
HTML 内容写入编辑器，并设置 initialContentLoaded 标记。

协同文档初始内容装载过程如下：

输入：数据库文档 HTML、Y.Doc 的 default
fragment、config.initialContentLoaded 标记。

输出：只初始化一次的协同文档内容。

步骤1：客户端等待协同连接完成同步后，读取 Y.Doc 中的 default fragment。

步骤2：如果 fragment
已有内容，说明协同房间中已经存在文档状态，当前客户端不再写入数据库
HTML。

步骤3：如果 config.initialContentLoaded 已为
true，说明已有客户端完成初始化，当前客户端只接收现有 CRDT 状态。

步骤4：只有当 fragment 为空、initialContentLoaded 不存在且数据库 HTML
非空时，客户端才将 HTML 写入编辑器，并把 initialContentLoaded 设置为
true。

该算法用于解决"多客户端同时进入空协作房间"时可能出现的重复 setContent
问题，保证数据库内容只作为协同文档的初始种子，而不会覆盖已经产生的实时编辑结果。

除正文内容外，系统还通过 Awareness 机制同步协作者状态。Awareness
数据只表示当前在线状态，不写入 versionHistory 和
updateHistory。系统会遍历 awareness states，过滤当前用户本人，并按
userId 合并同一用户的多个连接状态；当某个状态带有 cursor head
时，优先保留该状态。

协作者状态合并过程如下：

输入：Awareness states、当前用户 ID。

输出：按用户去重后的远端协作者列表。

步骤1：遍历 awareness.getStates() 返回的所有在线状态。

步骤2：过滤没有 user 信息的状态，并过滤当前用户自己的状态。

步骤3：以 userId 为键建立
Map，将同一用户的多个浏览器连接合并为一条显示记录。

步骤4：如果同一用户存在多个连接状态，优先保留带有 cursor.head
的状态，因为它能定位该用户正在编辑的块。

步骤5：将 Map 转换为数组，交给编辑器画布根据 cursor head
计算块级头像位置。

Awareness 数据不写入 versionHistory 和
updateHistory，只表示在线状态。这样在线用户变化不会污染文档历史，也减少了协同编辑时的无效持久化。

## 5.6 版本控制与更新记录实现

版本控制模块由 versionHistory、updateHistory
和前端历史面板共同组成。文档内容保存时，服务端 mutation
会先比较旧标题、旧正文与新标题、新正文。如果标题或正文发生变化，系统会向
updateHistory
追加一条更新记录，记录修改时间、修改用户、修改前后的标题和正文内容。

文档保存、权限校验与历史记录生成的关键代码如下：

const permission = getDocumentPermissionForUser(state, userId,
document);

if (typeof input.title === \"string\" && permission !== \"owner\") {

return { ok: false, error: \"Only owner can edit title\" };

}

if (typeof input.content === \"string\" && permission === \"can_view\")
{

return { ok: false, error: \"No edit permission\" };

}

const updateHistory = appendDocumentUpdate(document, {

editedAt, nextContent, nextTitle, userId,

});

const versionHistory = resolveVersionHistory({

contentChanged, document, editedAt,

mode: input.versionHistoryMode ?? \"merge\",

nextContent, nextTitle, userId,

});

上述代码展示了文档保存入口同时承担权限控制、更新记录和版本快照维护三项职责。标题修改只允许
owner 执行，正文修改拒绝 can_view 用户；通过校验后，系统再分别追加
updateHistory 并按保存模式维护 versionHistory。

版本快照生成过程如下：

输入：旧文档、保存后的新标题与新正文、versionHistoryMode、编辑时间和编辑用户。

输出：更新后的 versionHistory。

步骤1：判断正文是否发生有效变化。如果 HTML
完全相同，或去除标签后的正文为空，则不生成新版本。

步骤2：当 versionHistoryMode 为 snapshot
时，强制把当前标题和正文作为新快照写入 versionHistory。

步骤3：当 mode 为 merge
时，先检查最近一个版本快照的创建时间；如果仍处于合并窗口内，则不重复新增快照，避免连续输入造成大量历史版本。

步骤4：当 mode 为 force 或超过合并窗口时，系统创建新的 version\_{uuid}
快照，并放到版本列表头部。

步骤5：如果新快照与最近快照标题和正文完全相同，或历史中已经存在相同内容，则直接复用原版本列表。

该算法把"实时保存"和"可恢复版本"分开处理：自动保存可以频繁执行，版本快照则按模式和时间窗口控制数量，从而兼顾数据安全和历史列表可读性。

版本历史面板展示版本列表、版本创建时间和创建用户，用户选中某个版本后，系统会将该版本与前一个版本进行差异比较，并在预览区标记新增或删除内容。如图5-2所示，恢复历史版本并不是绕过保存逻辑直接覆盖数据库，而是以
versionHistoryMode: force
重新提交选中版本的正文内容，服务端仍会执行权限校验和状态更新。

Updates 差异计算的关键代码如下：

while (previousIndex \< previousTokens.length &&

currentIndex \< currentTokens.length) {

if (previousTokens\[previousIndex\] === currentTokens\[currentIndex\]) {

appendStatus(parts, \"unchanged\", previousTokens\[previousIndex\]);

previousIndex += 1;

currentIndex += 1;

} else if (lengths\[previousIndex + 1\]\[currentIndex\] \>=

lengths\[previousIndex\]\[currentIndex + 1\]) {

appendStatus(parts, \"removed\", previousTokens\[previousIndex++\]);

} else {

appendStatus(parts, \"added\", currentTokens\[currentIndex++\]);

}

}

上述代码展示了 Updates 面板会先把历史 HTML 转换为文本
token，再根据最长公共子序列结果判断文本片段属于未变化、新增还是删除。该处理方式避免直接展示整篇历史正文，使用户能够快速看到最近修改的具体内容。

Updates 文本差异生成过程如下：

输入：修改前 HTML、修改后 HTML、标题文本和图片/目录等特殊节点标签。

输出：新增、删除和未变化的文本片段序列。

步骤1：将前后两个 HTML 片段解析为
DOM，按标题、段落、列表、代码块、表格、图片等节点转换为可比较的纯文本。

步骤2：对纯文本进行 token 化处理，得到 previousTokens 和 currentTokens。

步骤3：构造 LCS 动态规划表 lengths，其中 lengths\[i\]\[j\] 表示
previousTokens 从 i 开始、currentTokens 从 j 开始的最长公共子序列长度。

步骤4：从两个 token 序列头部开始遍历。若当前 token 相同，则标记为
unchanged；若删除方向的 LCS 长度更大或相等，则标记 previousTokens\[i\]
为 removed；否则标记 currentTokens\[j\] 为 added。

步骤5：合并相邻同类型片段，并过滤空白变化，最终在 Updates
面板中以新增和删除样式展示。

该算法的时间复杂度为 O(mn)，其中 m 和 n 分别为修改前后 token 数。由于
Updates
面板针对单条修改记录计算差异，且系统只展示有限历史记录，因此该复杂度能够满足毕业设计场景下的交互需求。

## 5.7 文件与资源管理实现

媒体资源管理模块采用"应用路径统一、存储后端可替换"的设计。用户在编辑器中粘贴或拖放图片时，前端首先识别剪贴板或拖拽事件中的图片文件，然后调用上传逻辑将文件发送至服务端。服务端根据
STORAGE_BACKEND 配置选择本地文件系统或 S3 兼容对象存储。无论底层使用本地
.data/media 还是 MinIO，返回给编辑器的地址都保持为 /api/media/\...。

## 5.8 Markdown 导入导出实现

Markdown
导入导出模块用于解决文档迁移和离线备份问题。导入时，系统读取用户选择的
.md 或 .zip 文件，将 Markdown 文本转换为编辑器可识别的 HTML
结构，并把图片资源恢复为媒体路径后写入编辑器；导出时，系统从当前文档
HTML 中提取结构化内容，转换为 Markdown
文本，并在需要时将图片资源一起打包为 ZIP

## 5.9 数据存储实现

Syncdown 的应用状态存储支持 PostgreSQL 和本地 JSON 两种模式。当环境变量
DATABASE_URL
存在时，系统将用户、工作区、文档、权限、访问记录、版本历史等状态持久化到
PostgreSQL 的 syncdown_state 表中；当 DATABASE_URL 不存在时，则回退到
.data/app-state.json。该设计使系统既能在开发环境中快速启动，也能在接近生产环境的配置下使用数据库持久化。^\[10\]^

文档保存时，前端通过 PATCH /api/documents/{id}
提交标题或正文变更。服务端首先读取当前状态，然后根据当前用户和目标文档计算权限。只有
workspace owner 可以修改文档标题；owner 和 can_edit
用户可以修改正文；can_view 用户只能读取文档，不能提交正文变更。

## 5.10 AI 多模型辅助功能实现

AI 辅助功能由前端选区气泡、服务端代理接口和候选结果回填三部分组成。用户在编辑器中选中文本后，前端记录选区内容和操作类型，并向 /api/ai/action 发送请求。请求体包含 selectedText、action、locale、prompt 和 candidateCount 等字段。服务端读取 AI_API_KEY、AI_BASE_URL、AI_MODEL 和 AI_SECONDARY_MODEL 等环境变量，以 OpenAI 兼容协议请求模型服务。

系统在服务端完成模型列表去重和并发调用。当 candidateCount 为 1 时，仅调用默认模型；当 candidateCount 为 2 时，同时调用默认模型和备选模型，并将模型名称与生成结果一起返回前端。前端只负责展示候选结果，不接触模型密钥，从而降低密钥泄露风险。

AI 多模型候选结果生成的关键代码如下：

```ts
const models = Array.from(
  new Set([model, candidateCount === 2 ? secondaryModel : null].filter(Boolean)),
);

const candidates = (
  await Promise.all(
    models.map((candidateModel) =>
      requestAiCandidate({
        apiKey,
        baseUrl,
        locale: body.locale,
        model: candidateModel,
        payload: body,
      }),
    ),
  )
).filter((candidate) => candidate !== null);
```

上述代码表明，多模型对比并不是在浏览器中分别调用模型，而是在服务端统一完成。这样既能隐藏 API Key，也便于对超时、错误和空结果进行集中处理。当前端收到 candidates 后，会在 AI 结果面板中展示不同模型名称和生成内容。用户选择某个候选结果后，可以执行“替换选区”或“插入下方”。这两个操作都会调用编辑器命令修改 Tiptap 文档，并进一步由 Yjs 转换为 CRDT update，同步给同一文档房间内的其他协作者。

从工程边界看，AI 功能不直接修改数据库，也不绕过编辑器写入正文。它只生成候选文本，最终是否进入文档由用户确认。该实现方式能够兼顾 AI 大模型的智能生成能力与协作文档系统的数据安全性。

## 7.2 不足与改进

（2）增强版本控制能力，补充版本命名、块级差异展示和历史版本检索功能；

（3）引入更精细的权限控制模型，提高系统安全性与管理能力；

（4）完善 AI
